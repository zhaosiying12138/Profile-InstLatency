RVV scheduling model profiling

从 profile 到 LLVM RISC-V RVV 调度模型

本文说明实现一个 LLVM 调度模型需要 profile 哪些公共参数和指令参数，并用 `vdivu.vv` 作为代表，把汇编实验、marker delta、`profile.yaml`、`mismatch_report.md` 和 TableGen 草案串成一条可审计证据链。

研究背景与证据边界

LLVM 的 scheduling model 不是一张“指令延迟表”。它同时描述 issue 宽度、in-order/OOO 假设、执行资源、资源占用、生产者到消费者的可见时间，以及少数 issue group 约束。对于 RVV 指令，LMUL 会改变一个逻辑指令覆盖的元素组规模，因此同一 opcode 还需要分 LMUL profile。

本仓库有两层证据。第一层是 `real_platform_profile`：gem5 MinorCPU 跑真实 trace，`results/common/experiment_quality.md` 当前 gate 为 PASS，用来证明 profile 流水线有覆盖、重复性和人工风险接受。第二层是 `synthetic_calibration`：`results//profile.yaml` 作为可直接导出 `results/common/llvm_model_draft.td` 的输入，`results/common/mismatch_report.md` 显示所有 claimed raw-inferred 字段与配置真值一致。本文的 TableGen 数值以第二层为准，避免把尚未导出到 draft 的 sidecar 结果混入 LLVM 实现段。

代表指令 `vdivu.vv` / `WriteVIDivV`。 校准状态 `mismatch_report.md`: Gate status PASS, claimed mismatches none。 方法边界 图像只辅助理解，数值结论只来自代码块和表格。

IssueWidth

`IssueWidth` 是 LLVM scheduler 在同一个 cycle group 内最多能放入的 micro-op 数。它不是某条指令的延迟，而是整个处理器模型的公共上限。

当前公共 profile 把模型设为双发射：

processor:
  issue_width:
    value: 2
    source: prior_from_task_statement
    evidence:
      - docs/plan.md

用于观察 issue group 行为的汇编形态是 T21，向量指令后接 scalar `add`。如果 vector 指令能够和 scalar companion 同组，说明没有额外 single-issue 压力；如果持续阻塞，则要考虑 `NumMicroOps` 或 group flag。

__rvv_profile_marker_t21_vdivu_vv_m1_n4_start:
    vdivu.vv v2, v0, v1
    add x20, x21, x22
    vdivu.vv v3, v0, v1
    add x23, x24, x25
    vdivu.vv v4, v0, v1
    add x26, x27, x28
    vdivu.vv v5, v0, v1
    add x29, x30, x31
__rvv_profile_marker_t21_vdivu_vv_m1_n4_end:
`t21-vdivu-vv-m1-n4/trace.json`: start 1000, end 1024, primary delta 24。对 `profile.yaml` 校准链而言，这一数据不单独 claim `IssueWidth`，公共字段由 `processor.yaml` 固定。

TableGen 草案中的实现是机器模型字段：

def YuShuXinModel : SchedMachineModel {
  let IssueWidth = 2;
  let MicroOpBufferSize = 0;
  let LoadLatency = 0;
}

MicroOpBufferSize

`MicroOpBufferSize` 表示 LLVM 对 issue 与 scheduling 之间缓冲能力的假设。`0` 或 `1` 是 in-order 风格；大于 `1` 时，LLVM 会把模型视作有 OOO buffer。

processor:
  micro_op_buffer_size:
    value: 0
    source: prior_from_task_statement
  out_of_order:
    value: false
  rob_entries:
    value: 0

在这种假设下，T10/T11/T12 的解释都按程序顺序构建：独立流看资源吞吐，自依赖链看 RAW readiness，而不是让 OOO issue 重排掩盖 stall。

# T11 self RAW chain: order is the measurement surface.
__rvv_profile_marker_t11_vdivu_vv_m1_n4_start:
    vdivu.vv v2, v0, v1
    vdivu.vv v2, v2, v1
    vdivu.vv v2, v2, v1
    vdivu.vv v2, v2, v1
__rvv_profile_marker_t11_vdivu_vv_m1_n4_end:
`t11-vdivu-vv-m1-n4/trace.json`: start 1000, end 1072, delta 72。由于是四条 RAW chain，校准 profile 推出 `Latency = 18`。

LLVM 实现仍落在同一个 `SchedMachineModel`：

let MicroOpBufferSize = 0;

ProcResource / ProcResGroup

`ProcResource` 是资源本体，`ProcResGroup` 是可任选其一的资源组。RVV 指令若只能进入某条 pipe，就用单个 `ProcResource`；若能在两条 pipe 间调度，就用 group。

processor:
  rvv_pipelines:
    count:
      value: 2
    resources:
      - name: YuShuXinVPipe0
      - name: YuShuXinVPipe1
      - name: YuShuXinAnyVPipe
        kind: proc_resource_group
        members: [YuShuXinVPipe0, YuShuXinVPipe1]

`vdivu.vv` 在校准 profile 中被归到 `YuShuXinVPipe1`。T20 pairwise 实验把 `vdivu.vv` 和另一个指令交错，观察 pair slope 是否出现同资源串行。

__rvv_profile_marker_t20_vdivu_vv_vredsum_vs_m1_n4_start:
    vdivu.vv  v2, v0, v1
    vredsum.vs v3, v0, v1
    vdivu.vv  v4, v0, v1
    vredsum.vs v5, v0, v1
    vdivu.vv  v6, v0, v1
    vredsum.vs v7, v0, v1
    vdivu.vv  v8, v0, v1
    vredsum.vs v9, v0, v1
__rvv_profile_marker_t20_vdivu_vv_vredsum_vs_m1_n4_end:
`t20-vdivu-vv-vredsum-vs-m1-n4/trace.json`: start 1000, end 1032, delta 32。`profile.yaml` 的 `m1.proc_resource.value` 为 `YuShuXinVPipe1`，confidence 为 `raw_marker_pairwise_inference`。 
def YuShuXinVPipe0 : ProcResource<1>;
def YuShuXinVPipe1 : ProcResource<1>;
def YuShuXinAnyVPipe : ProcResGroup<[YuShuXinVPipe0,
                                      YuShuXinVPipe1]>;

Latency

`Latency` 是 LLVM scheduling model 里最核心的指令字段之一。它表示 producer 写结果后，dependent consumer 何时可以读取这个结果。它不是独立指令吞吐，也不等同于物理 writeback stage。

校准 profile 对 `vdivu.vv` 的结论如下：

LMULLatency证据
`m1`18`t11-vdivu-vv-m1-n4`: delta 72 / 4
`m2`24`t30-vdivu-vv-t12-m2-k0`: readiness 24
`m4`36`t30-vdivu-vv-t12-m4-k0`: readiness 36
 
__rvv_profile_marker_t30_vdivu_vv_t12_m1_k0_start:
    vdivu.vv v2, v0, v1
    vadd.vv  v3, v2, v1
__rvv_profile_marker_t30_vdivu_vv_t12_m1_k0_end:
{
  "experiment_id": "t30-vdivu-vv-t12-m1-k0",
  "observation": {
    "primary_delta_cycles": 18,
    "filler_count": 0,
    "template_id": "T30_LMUL_SCALING"
  },
  "synthetic": {
    "latency_cycles": 18,
    "release_cycles": 6,
    "pipe": "pipe1"
  }
}

LLVM 实现时，`Latency` 写入对应 LMUL 的 `SchedWriteRes`：

def YuShuXinWriteVIDivV_M1 : SchedWriteRes<[YuShuXinVPipe1]> {
  let Latency = 18;
  let ReleaseAtCycles = [6];
}

ReleaseAtCycles

`ReleaseAtCycles` 描述资源占用持续时间。长 latency 的指令可以只占用执行 pipe 很短时间；反过来，吞吐差的指令也可能资源占用很长。LLVM 通过它建模独立指令流的 steady-state slope。

__rvv_profile_marker_t10_vdivu_vv_m1_n4_start:
    vdivu.vv v2, v0, v1
    vdivu.vv v3, v0, v1
    vdivu.vv v4, v0, v1
    vdivu.vv v5, v0, v1
__rvv_profile_marker_t10_vdivu_vv_m1_n4_end:
{
  "experiment_id": "t10-vdivu-vv-m1-n4",
  "observation": {
    "iterations": 4,
    "primary_delta_cycles": 24,
    "template_id": "T10_INDEPENDENT_STREAM_THROUGHPUT"
  },
  "synthetic": {
    "release_cycles": 6
  }
}
LMULReleaseAtCycles独立流解释
`m1`[6]四条独立 `vdivu.vv` delta 24。
`m2`[8]资源占用随 LMUL 增长。
`m4`[12]更大的 LMUL 使 pipe 更久不能接收同类指令。

LLVM 中与 `Latency` 同处一个 `SchedWriteRes`：

let ReleaseAtCycles = [6];

AcquireAtCycles

`AcquireAtCycles` 与 `ReleaseAtCycles` 成对出现，用来描述资源不是在 issue 当下立即获取，而是在 issue 后若干 cycle 才开始占用。默认空列表表示资源从 issue 起占用。

`vdivu.vv` 当前校准 profile 对该字段不 claim，保持默认空列表：

acquire_at_cycles:
  value: []
  confidence: "assumed_default"
  claimed: false
  identifiable: true
  evidence: []

对应的汇编仍是 T10/T20 一类资源实验；只是当前 marker 观测不要求“延迟获取”这一额外自由度，因而 TableGen 草案不写 `AcquireAtCycles`。

结论：不生成 LLVM 字段。只有当不同 pair/throughput 形态无法被 issue-time acquisition 解释时，才需要把 `AcquireAtCycles` 写入 `SchedWriteRes`。

NumMicroOps

`NumMicroOps` 决定一条调度写对 `IssueWidth` 的压力。如果一条指令展开为多个 scheduler micro-op，即使资源没有冲突，也可能占满 issue group。

校准 profile 中，`vdivu.vv` 记录了这个字段，但没有 claim：

num_micro_ops:
  value: null
  confidence: "not_identifiable"
  claimed: false
  identifiable: false
  reason: "T20/T21 marker evidence is not sufficient to decompose scheduler writes into micro-ops."

它的 profile 汇编仍来自 T21 scalar pairing：

vdivu.vv v2, v0, v1
add x20, x21, x22
vdivu.vv v3, v0, v1
add x23, x24, x25
`t21-vdivu-vv-m1-n4/trace.json`: delta 24。该 delta 能参与候选约束，但 `profile.yaml` 不把它升级成 `NumMicroOps` 的 claimed LLVM 字段；`llvm_model_draft.td` 也只留下注释。 
// NumMicroOps is recorded in the profile but not claimed by current evidence.

SingleIssue / BeginGroup / EndGroup

`SingleIssue` 表示该指令必须独占一个 issue group；`BeginGroup` 和 `EndGroup` 则分别约束 group 边界。它们影响的是 group packing，不是执行 latency。

当前校准 profile 对 `vdivu.vv` 不 claim 这些字段：

single_issue:
  value: null
  confidence: "not_identifiable"
  claimed: false
  identifiable: false
  reason: "Scalar-pairing evidence is not sufficient to claim SingleIssue."

同样使用 T21 汇编作为 probe，但结果没有达到“必须独占 group”的 claim 门槛。

__rvv_profile_marker_t21_vdivu_vv_m1_n4_start:
    vdivu.vv v2, v0, v1
    add x20, x21, x22
    vdivu.vv v3, v0, v1
    add x23, x24, x25
__rvv_profile_marker_t21_vdivu_vv_m1_n4_end:
LLVM 实现：不写 `let SingleIssue = 1`，不写 `BeginGroup`，不写 `EndGroup`。保持默认比误建模更安全。 
// SingleIssue is recorded in the profile but not claimed by current evidence.

ReadAdvance

`ReadAdvance` 是 producer-consumer pair 的修正项。当某类 consumer 通过 bypass 比默认 `Latency` 更早读到结果，LLVM 可用 `SchedReadAdvance` 表示这个差值。

`vdivu.vv` 当前没有 consumer-specific bypass claim：

read_advance:
  value: null
  confidence: "not_identifiable"
  claimed: false
  identifiable: false

T12/T30 的汇编只证明 `vadd.vv` 这个 consumer 的 readiness；没有比较出另一类 consumer 的不同 readiness，因此不能写 `ReadAdvance`。

vdivu.vv v2, v0, v1
vadd.vv  v3, v2, v1
LLVM 实现：不生成 `ReadAdvance`。如果后续加入不同 read class 的 consumer-gap 实验，并观察到稳定提前量，再补 `SchedReadAdvance`。

LMUL scaling

RVV 调度模型不能只给 opcode 一个标量值。`m1/m2/m4` 改变向量寄存器组规模，也改变 latency 和 resource occupancy。校准 profile 对 `vdivu.vv` 给出精确线性拟合：

字段公式`m1``m2``m4`
`Latency``12 + 6 * LMUL`182436
`ReleaseAtCycles``4 + 2 * LMUL`6812
 
fit:
  latency_formula:
    form: base_plus_lmul_times_k
    base: 12
    k: 6
    residual: 0
    status: exact_fit
  release_formula:
    form: base_plus_lmul_times_k
    base: 4
    k: 2
    residual: 0
    status: exact_fit

LLVM TableGen 草案选择按 LMUL 拆成离散 `SchedWriteRes`，避免在最终文件里留下需要读者再次解释的公式占位。

timestamp_model

`timestamp_model` 不是 LLVM 调度字段，但它是 profile 方法论的根基。marker 如果占 issue slot 或 functional unit，会污染所有 delta。

timestamp_model:
  kind: zero_cost_label_marker
  occupies_issue_slot: false
  cycles: 0
  evidence:
    - results/common/experiments/t00-marker/trace.json
    - scripts/run_experiment.py

所有关键汇编片段都使用同一约定：marker 是 label-PC wrapper，不生成实际 RVV 指令。

.globl __rvv_profile_marker_t10_vdivu_vv_m1_n4_start
__rvv_profile_marker_t10_vdivu_vv_m1_n4_start:
    vdivu.vv v2, v0, v1
    ...
.globl __rvv_profile_marker_t10_vdivu_vv_m1_n4_end
__rvv_profile_marker_t10_vdivu_vv_m1_n4_end:
marker baseline cycles 为 0；相邻 marker 不改变 issue、scoreboard 或资源使用。

TableGen 草案实现步骤

把 profile 结果写进 LLVM 时，先建立机器模型和资源，再为目标 `SchedWrite` family 按 LMUL 写 `SchedWriteRes`。当前草案片段如下：

def YuShuXinModel : SchedMachineModel {
  let IssueWidth = 2;
  let MicroOpBufferSize = 0;
  let LoadLatency = 0;
}

def YuShuXinVPipe0 : ProcResource<1>;
def YuShuXinVPipe1 : ProcResource<1>;
def YuShuXinAnyVPipe : ProcResGroup<[YuShuXinVPipe0, YuShuXinVPipe1]>;

// Evidence: t30-vdivu-vv-t12-m1-k0 @ results/vdivu_vv/experiments/t30-vdivu-vv-t12-m1-k0/trace.json
def YuShuXinWriteVIDivV_M1 : SchedWriteRes<[YuShuXinVPipe1]> {
  let Latency = 18;
  let ReleaseAtCycles = [6];
  // NumMicroOps is recorded in the profile but not claimed by current evidence.
  // SingleIssue is recorded in the profile but not claimed by current evidence.
}

def YuShuXinWriteVIDivV_M2 : SchedWriteRes<[YuShuXinVPipe1]> {
  let Latency = 24;
  let ReleaseAtCycles = [8];
}

def YuShuXinWriteVIDivV_M4 : SchedWriteRes<[YuShuXinVPipe1]> {
  let Latency = 36;
  let ReleaseAtCycles = [12];
}

落到真正的 LLVM 后，还需要把 `WriteVIDivV` family 的 LMUL/SEW variant 连接到这些目标写资源。本文只覆盖“如何表达 profile 结果”的层级，不展开完整 upstream patch。

结论与非声明范围

实现 LLVM RVV 调度模型时，必须先把公共处理器参数和每类指令参数分开。公共层决定 scheduler 的全局约束；指令层决定具体 `SchedWriteRes`。`vdivu.vv` 的校准链给出清晰的 `ProcResource`、`Latency`、`ReleaseAtCycles` 与 LMUL scaling，并且 `mismatch_report.md` 对 claimed 字段给出 PASS。

同样重要的是不生成缺证据字段。`AcquireAtCycles`、`NumMicroOps`、`SingleIssue/BeginGroup/EndGroup`、`ReadAdvance` 在当前 `profile.yaml` 到 `llvm_model_draft.td` 的链路中没有被 claim；因此 TableGen 草案保留默认值或注释，而不是写入看似完整但不可证的字段。

最终可写入的核心结构是：`IssueWidth = 2`、`MicroOpBufferSize = 0`、`YuShuXinVPipe0/1`、`YuShuXinAnyVPipe`，以及 `YuShuXinWriteVIDivV_M1/M2/M4` 的 latency 和 release values。
